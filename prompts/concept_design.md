# 概念想法到创新方案设计（Step 2）

## 目标

把用户输入的一段概念想法，扩展为可专利化、可实施、可查新的技术方案底稿。该步骤替代旧版「必须先扫描项目」的默认入口；仅当用户另给项目路径或附件时，才用 `project_scan.md` 补充事实。

## 输入类型

用户可能只提供：

- 一句话想法：如「用大模型帮客服自动发现高风险投诉」
- 产品/业务痛点：如「现场设备太多，故障定位慢」
- 技术方向：如「边缘端隐私计算」「多模态质检」
- 零散功能清单：如「采集日志、预测异常、自动派单」

不要因为缺少代码、PRD 或实验数据而终止。信息不足时先做合理技术假设，并把假设标明为「待确认」。

## 设计原则

- 先把概念转成「技术问题」而不是营销卖点。
- 创新点必须落在技术手段、数据处理、控制流程、模型结构、参数规则、系统协同或异常处理上。
- 可推演工程实现，但不得编造用户未提供的真实指标、上线事实、客户名称、实验结果或内部系统名称。
- 优先设计「方法 + 系统/装置 + 存储介质」均可承接的方案。
- 避免只写抽象 AI 套壳。若涉及大模型、机器学习或自动化决策，必须说明输入、特征/状态、处理链路、约束规则、输出与闭环反馈。

## 执行步骤

### 2.1 概念拆解

用 5–8 行梳理：

- 应用场景
- 当前痛点
- 处理对象或数据来源
- 目标输出
- 关键约束（实时性、隐私、成本、准确性、可解释性、资源限制等）
- 用户已明确的事实
- 需要假设的空白

### 2.2 技术问题重写

把概念改写成 1–3 个专利语境下的技术问题。句式建议：

```
如何在[约束/场景]下，对[对象/数据]进行[处理动作]，以解决[现有技术缺陷]的问题。
```

### 2.3 方案骨架设计

设计一个完整技术链路，至少包含：

- 数据/信号采集或输入归一化
- 状态/特征/上下文建模
- 核心判断、生成、调度、控制或优化机制
- 结果输出与反馈闭环
- 异常、低置信度或冲突场景处理
- 可选部署形态：端侧、云侧、边云协同、插件、服务端流程等

如适用，补充 1 个轻量公式或评分规则，但只在确有必要时使用。

### 2.4 候选创新点设计

生成 3–5 个候选创新点。每个候选点包含：

- 名称：一句专利式标题
- 技术手段：不是效果口号，要写清处理链路或机制
- 要解决的技术问题
- 区别于常规方案的可能差异
- 可实施要素：模块、步骤、数据结构、规则或模型输入输出
- 风险：哪些部分可能被现有技术覆盖，查新时重点检索什么

### 2.5 组合方案与推荐

判断候选点是否应合并为一篇：

- 若多个创新点共享同一输入、状态模型或闭环流程，优先组合成一篇「方法及系统」。
- 若候选点分别对应不同产品/不同技术效果，建议拆成多篇并说明顺序。
- 默认推荐最适合先写的一篇，并给出 1 个案件名称草案。

### 2.6 待确认问题

最多问 3 个高价值问题。不要把所有细节都抛给用户。若用户要求直接生成，则用假设继续：

- 关键数据来源是否真实存在？
- 核心处理发生在端侧、云侧还是边云协同？
- 更希望保护算法流程、系统架构、交互闭环还是部署机制？

## 输出格式

按以下结构输出给后续步骤使用：

```markdown
## 概念理解
- ...

## 技术问题
1. ...

## 方案骨架
1. ...

## 候选创新点
### 候选点 1：...
- 技术手段：
- 技术问题：
- 可能差异：
- 可实施要素：
- 查新风险：

## 推荐成案方向
- 推荐案件名称：
- 推荐理由：
- 合并/拆分判断：

## 待确认假设
- ...
```

后续 `patent_points_analyzer.md`、`prior_art_search.md`、`disclosure_preview.md` 均以本输出作为主要材料。
